## Part B — New Findings (Slideshow + recent commits)

- `src/pages/Films.jsx` (cascade orchestration)
- `src/components/layout/Navigation.jsx` (mobile nav fix)

### Findings

**N1 (HIGH) — Slideshow batch-preloads all 116 photos at mount**
`FloatingGalleryHero.jsx:128-132` calls `new Image()` for every entry in `queueRef.current` synchronously on effect run. That fires ~116 simultaneous network requests on `/photos` page load. Spec says "preload the next 3 in the queue" — current code does the opposite extreme. Bandwidth/network cost, especially on mobile 4G. The aggressive prefetch undermines the cascade-load philosophy used on the Films page.
**Fix:** preload only the next ~3 in the rolling window.

**N2 (HIGH) — `isMobile` captured once, never re-evaluated**
`FloatingGalleryHero.jsx:82` reads `window.innerWidth` at hook mount. The `useEffect` deps (line 160) are `[advance, prefersReducedMotion]`, so resizing past 768 (rotation, devtools) doesn't restart with the new filter. Landscape-skip stays in its initial state.
**Fix:** track via `matchMedia` listener or include `isMobile` in effect deps.

**N3 (MEDIUM) — Slideshow file 372 LOC > 300 LOC quality gate**
`FloatingGalleryHero.jsx` — extract `usePhotoSlideshow`, `NameOverlay`, and `MobileHeroNameOverlay` (already imported from HeroSection but a sibling exists at line 336) into separate files.

**N4 (MEDIUM) — `prefers-reduced-motion` path skips brightness analysis**
When reduced-motion is on, the effect returns early (line 125) without ever running `preloadImage` to compute `isLight`. The first photo renders with default `isLight=false`, so name overlay always uses light-mode color logic. Functional but visually wrong on bright reduced-motion photos.
**Fix:** still run a one-shot `preloadImage(initial[0])` to set brightness even when cycling is disabled.

**N5 (MEDIUM) — `advance()` silently fails after 20 attempts**
`FloatingGalleryHero.jsx:104-121` caps the search at 20 attempts. If a portrait-heavy queue has a stretch of >20 landscape on mobile, `advance` returns without calling `setCurrentPhoto`, but the outer `tick` schedules another timer — so the user sees a frozen frame for one cycle. Rare in practice given current photo set, but invisible failure.
**Fix:** filter `heroPhotos` once for portrait-only on mobile rather than reject-and-retry.

**N6 (MEDIUM) — Mobile name overlay uses fixed white color**
Line 336 renders `MobileHeroNameOverlay` (white) regardless of photo brightness. On bright photos this is low-contrast vs. the desktop adaptive black/white logic. Accessibility regression on mobile.
**Fix:** thread `currentPhoto.isLight` into mobile overlay and switch color.

**N7 (LOW) — Mobile vs. desktop hero height inconsistent**
Desktop wrapper uses `100svh - 5rem + 150px` (intentional per memory). Mobile path lacks the `+150px` since it doesn't share the desktop sticky behaviour. The Films hero applies the same 150px across both — verify if Photos mobile should match.

**N8 (LOW) — heroPhotos.js percent-encoded paths**
Filenames already contain `%20` and `%C3%A9`. If a future file is added without pre-encoding, the React `<img src>` will not auto-encode. No defensive helper exists; relies on author convention.

**N9 (LOW) — Films cascade resets on every mount, not on navigation**
`Films.jsx:36` `featuredLoadCount = useRef(0)` resets on remount. Fine in current SPA flow, but if Films is routed with key persistence in future, count survives stale.

**N10 (INFO/CLOSED) — "Next 3 preload" spec deviation**
`.mdd/docs/01-photos-hero-slideshow.md:35` mandates preloading the next 3. Implementation preloads ALL — contradicts spec. Either tighten code (preferred — see N1) or update doc to reflect "all photos preloaded".

**N11 (LOW) — Duplicate `MobileHeroNameOverlay` source of truth**
Imported from `HeroSection.jsx` (`FloatingGalleryHero.jsx:13`) but the component is also re-implemented locally. Verify no fork; consolidate.

### [07] CMS Foundation
**Files read:** sanity.config.js, sanity/schemas/index.js, siteSettings.js,
heroOverlay.js, showreel.js, sanity/desk/structure.js, sanity/tools/DeployTool.jsx,
src/sanity/client.js, src/sanity/loader.js, src/pages/Studio.jsx,
scripts/fetch-cms-content.mjs, scripts/seed-cms.mjs, scripts/watch-cms.mjs,
vercel.json, scripts/vercel-build.sh, .env.example, package.json

**Findings:**
- 🟠 Doc 07 Data Model still documents `heroOverlayItem` with live `offsetX`/
  `offsetY` positioning fields + `stackRowGap` (default 24) and a 9-anchor enum
  including center anchors. Reality: offsets retired (`hidden:true` in schema),
  positioning is fixed insets (doc 14), anchor enum has only 6 options (no
  center). Business Rules 8–9 (offset positioning, `fluidScale mobileRatio 0.4`,
  `STACK_VIEWPORT_PADDING`) are superseded by docs 13/14. No supersession note.
- 🟠 Doc 07 Business Rule 12 webhook GROQ filter omits `film`:
  `_type in ["siteSettings","heroOverlay","showreel","heroPhotos","photoProject"]`.
  CLAUDE.md's filter correctly includes `film`. Copying doc 07's version into
  the live webhook would make published film changes silently not deploy.
- 🟠 `seed-cms.mjs` is stale vs current schema: seeds `heroOverlay` items with
  retired `offsetX`/`offsetY`, omits `textSize` + `autoShrinkSmallScreens`;
  omits `navLinkSize`/`navLinkActiveSize` on siteSettings. `cms:seed` is the
  documented "reset to fresh" path — running it reverts the overlay to
  pre-doc-13/14 data.
- 🟠 `watch-cms.mjs` listen query is `*[_id in ["siteSettings","heroOverlay",
  "showreel"]]` — misses `heroPhotos`, `photoProject`, `film`. Editing those in
  Studio does not trigger a local refetch. CLAUDE.md advertises `cms:watch` as
  "Live-reload when Sanity content changes" with no caveat.
- 🟠 `src/sanity/client.js` is dead code — nothing imports the local module
  (fetcher/watcher/all scripts each construct their own `@sanity/client`). Its
  header comment claims "Used by the build-time fetcher" (false). `.env.example`
  also cites it as the project-ID source of truth.
- 🟡 `seed-cms.mjs` L127-129 references a Sanity webhook "Delay" setting —
  contradicts CLAUDE.md ("Sanity does NOT have a built-in webhook debounce").
- 🟡 `fetch-cms-content.mjs` header comment says "Reads the three singleton
  documents" — actually reads 4 singletons + 2 collections.
- 🟡 `Studio.jsx` comment says bundle is "~3MB gzipped"; CLAUDE.md + doc 07 say
  "~5MB / 1.7MB gzipped".
- 🟡 `fetch-cms-content.mjs` projects siteSettings/heroOverlay/heroPhotos as
  bare `*[..][0]` and showreel with `...`, so cms.json carries Sanity system
  fields (`_rev`, `_createdAt`, `_updatedAt`, `_system`). films/photoProjects
  use explicit field lists — inconsistent + bundle bloat.
- 🟡 `vercel-build` npm script is `vite build` only (skips CMS fetch). Dead
  because `vercel.json buildCommand` runs `vercel-build.sh`. If the buildCommand
  is ever removed, deploys silently ship stale CMS content.
- 🟡 Project config constants (`e9pgmdfm`, `production`, `2024-12-01`)
  hand-duplicated across ~10 files. Two singleton lists duplicated
  (`SINGLETON_TYPES` in sanity.config.js, `SINGLETONS` in structure.js).
- 🟢 Deploy tool, singleton enforcement (structure.js + action filter),
  fail-soft fetcher, code-split Studio — all correct and well-commented.

**Test coverage:** tests/e2e/cms-studio.spec.js (2 tests) — doc 07 frontmatter
says `test_files: []` (stale).
**Doc accuracy:** several stale sections (see above).

### [10] CMS Films
**Files read:** sanity/schemas/film.js, scripts/upload-films.mjs,
scripts/fix-film-ranks.mjs, fetch-cms-content.mjs (films branch),
loader.js (films exports), src/data/cms.json (films array)

**Findings:**
- 🔴(HIGH) `scripts/upload-films.mjs` writes 4 fields the current `film` schema
  no longer declares: `featured` (L103), `videoType` (L106), `videoFile` (L107),
  `imagePosition` (L113). `cms:upload-films` is a documented, supported
  re-import path (CLAUDE.md scripts table). Running it produces docs with 4
  orphan "unknown field" warnings in Studio and silently drops data into dead
  fields. The schema was rewritten (doc 12 Mux migration dropped `videoFile`/
  `videoType`; `featured` + `imagePosition` also removed) but the migration
  script was never updated. It also never writes the new `visible` field.
- 🔴(HIGH) Doc 10 Data Model + Business Rules are substantially out of sync with
  the shipped schema: still documents `featured`, `videoType`, `videoFile`,
  `imagePosition` as live fields; omits `videoMux` and `visible`. Business
  Rule 1 describes a featured/collapsed interplay that no longer exists (the
  homepage now alternates by row index — CLAUDE.md). Build-plan step 3 says the
  loader exports `getFeaturedFilms()` — actual loader exports
  `getNonCollapsedFilms()` / `getCollapsedFilms()`. `status` line still says
  "awaiting user-side run" though 17 films are live in cms.json.
  `known_issues` claims ESLint config is missing — `.eslintrc.cjs` exists.
- 🟢 `film` schema itself is clean (single-tab, fieldsets, Mux + Vimeo split,
  visible/collapsed, advanced overrides). `flattenFilm` correctly rebuilds
  `{left,right}` credits and resolves aspect ratio. cms.json confirms films
  carry `visible`/`collapsed`/mux fields and NO `featured` — fetcher + schema
  are mutually consistent; only the script + doc lag.
- 🟢 `fix-film-ranks.mjs` clean.

**Test coverage:** tests/e2e/cms-films-cdn.spec.js (2), cms-films-mux.spec.js (1).

## Cross-cutting summary

The Sanity *runtime* path (schemas → fetcher → loader → components) is healthy
and internally consistent — cms.json proves the live pipeline matches the
schemas. Every finding is **drift**: migration scripts and MDD docs that were
not updated when the `film` schema was rewritten (Mux migration + featured
removal) and when the hero overlay moved to fixed insets (docs 13/14).

- 1 HIGH code finding: `upload-films.mjs` schema drift.
- 1 HIGH doc finding: doc 10 out of date.
- "Webhook Delay" myth recurs in 3 scripts — coherent cluster, contradicts
  CLAUDE.md.
- No CRITICAL findings. The site builds and renders correctly.

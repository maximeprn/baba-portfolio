# MDD Audit Report — 2026-05-20

## Executive summary

The CMS migration (Stages 1, 3, 4 + Studio Deploy tool) introduced
~3000 lines of code across 30+ new files, but the MDD docs were only
partially updated. Doc 07 was written **before** Stages 3 and 4 and
hasn't been rewritten. Docs 01, 02, 05, 06 still reference the
legacy data files as the source of truth even though Photos.jsx and
FloatingGalleryHero.jsx now read from the CMS loader.

Net effect: a developer reading the docs today would think the site
is still backed by static JS data files, miss the entire CMS layer,
and have no idea how Studio editing makes its way to the live site.

## Findings by severity

### 🔴 CRITICAL (3)

1. **Doc 07 is severely outdated.**
   - Title says "Stage 1 of 6"; reality is Stages 1+3+4 all done.
   - `source_files` missing 9+ files including `heroPhotos.js`, `photoProject.js`, `DeployTool.jsx`, 4 of the 5 cms:* scripts, `fluidScale.js`, the updated Navigation.jsx, and the components now wired to the loader.
   - Data Model section documents 3 of 5 schemas; missing heroPhotos + photoProject.
   - No mention of `orderRank`/LexoRank/drag-to-reorder, the Sanity webhook gotchas, the Vercel deploy-hook dedupe behaviour, or the Studio Deploy tool.

2. **No MDD docs for Stages 3 & 4.**
   - Stage 3 (heroPhotos singleton + `cms:upload-hero-photos`) → 116 photos in CMS, a schema, a migration script, an integration into the live site — undocumented.
   - Stage 4 (photoProject collection + drag-reorder + `cms:upload-photo-projects` + `cms:fix-photo-project-ranks` + 22 doc migration) — even more substantial than Stage 3, also undocumented.

3. **Zero tests for any CMS feature.**
   - Doc 07's original test plan called for 4 Playwright specs (Studio loads, hero overlay, site copy, 404). Never written.
   - The Stage 4 photo-project flow has no integration test either, despite a flaky drag-to-reorder bug that took two debug rounds to resolve (zero-padded → LexoRank).

### 🟡 MEDIUM (5)

4. Docs 01, 02, 05, 06 still list legacy `src/data/*.js` files in `source_files` without indicating they're now fallbacks of the CMS loader.

5. Doc 02 (Design System) doesn't mention `src/utils/fluidScale.js` — a system-level helper used by both nav and hero text.

6. `src/data/siteConfig.js` still has several hardcoded fields (`tagline`, `shortBio`, `title`, `contact.*`, the navigation array) that look like they should be in CMS but weren't migrated. Not documented as a known gap.

7. Doc 05's `source_files` lists `scripts/scaffold-photo-projects.mjs` which is now obsolete (data lives in Sanity).

8. ## Decision points

- **One mega-doc 07 vs split into 07/08/09?** I lean toward splitting:
  - `07-cms-foundation` (Sanity setup + siteSettings + heroOverlay + showreel + Studio + Deploy tool)
  - `08-cms-hero-photos` (heroPhotos singleton + slideshow integration + upload script)
  - `09-cms-photo-projects` (photoProject collection + drag-reorder + LexoRank + upload + fix scripts + mobile-fallback fields)
  - Easier to find. Mirrors the migration stages. Smaller blast radius when one section ages.
- **Test coverage scope.** Even one Playwright e2e per CMS stage would be a leap from 0. We can stub: visit `/admin` and assert the Studio renders; visit `/photos` and assert the slideshow renders some `<img>` with a `cdn.sanity.io` src.

Estimated total fix time: **3 h doc work + 2 h test work = 5 h** (vs ~8–10 h if done piecemeal as new stages land).
